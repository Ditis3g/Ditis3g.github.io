# 블로그 메모 작성

사용자가 지금 진행한 대화/세션에서 배우거나 한 일을 한국어 **공부 메모**로
정리해 `D:\blog-memo` 폴더에 저장하는 스킬이다.

## 이 메모가 어디에 쓰이는지 (중요한 맥락)

저장한 메모는 **그 자체가 최종 블로그 글이 아니다.** 매일 밤 11시에 도는
자동화 스크립트(`D:\Project\Blog\scripts\daily-blog-draft.ps1`)가
`D:\blog-memo`의 메모들을 읽어, 클로드를 한 번 더 호출해 Hugo 블로그
**초안 글**로 다듬는다. 즉 이 메모는 그 글의 **재료(원본 노트)**다.

따라서 메모는 다음을 만족해야 한다:
- **사실이 정확해야 한다.** 뒤에서 글로 부풀려지므로, 여기서 틀리면
  블로그 글도 틀린다. 세션에서 실제로 다룬 내용만 적고, 추측·창작은 하지 마라.
- **핵심이 빠지지 않아야 한다.** 나중에 글로 풀어쓸 때 필요한 개념·이유·
  순서·구체적 값(명령어, 설정 이름 등)을 담아라.
- **장황할 필요는 없다.** 매끄러운 산문이 아니라, 요점이 잘 잡힌 노트면 된다.
  글로 다듬는 건 자동화가 한다.

## 작성 절차

1. **세션 내용을 돌아본다.** 이번 대화에서 사용자가 무엇을 배웠는지/했는지,
   어떤 문제를 풀었는지, 어떤 결정과 이유가 있었는지 추린다. 블로그 독자에게
   유익할 만한 "오늘의 배움" 관점으로 고른다. (잡담·인사·시행착오 중 의미
   없는 부분은 뺀다. 단, 겪은 함정과 해결법은 독자에게 유용하니 남긴다.)

2. **주제(topic)를 한두 단어로 정한다.** 파일 이름에 쓸 짧은 한국어 또는
   영어 키워드. 예: `github블로그`, `hugo-배포`, `docker기초`.

3. **메모 파일을 저장한다.** Write 도구로 아래 경로에 저장:

   ```
   D:\blog-memo\YYYY-MM-DD-<주제>.md
   ```

   - 날짜는 오늘 날짜(현재 세션 기준). 모르면 사용자에게 묻거나
     시스템의 현재 날짜를 사용한다.
   - 같은 이름이 이미 있으면 뒤에 `-2`, `-3`을 붙여 덮어쓰지 않는다.
   - 인코딩은 UTF-8. (Write 도구가 기본으로 UTF-8로 저장하므로 그대로 두면 됨.)

4. **저장 후 사용자에게 알린다.** 저장한 파일 경로와, "오늘 밤 11시 자동화가
   이걸 블로그 초안으로 올린다"는 점을 한 줄로 안내한다. 지금 바로 올리고
   싶어 하면, 다음을 실행하면 된다고 알려준다(또는 대신 실행해 준다):
   `powershell -ExecutionPolicy Bypass -File D:\Project\Blog\scripts\daily-blog-draft.ps1`

## 메모 형식

아래 틀을 기본으로 하되, 세션 내용에 맞게 유연하게 조정하라.

```markdown
# <오늘 다룬 주제 한 줄>

날짜: YYYY-MM-DD

## 오늘 배운 것 / 한 일
- 핵심 포인트를 항목으로. 개념은 "무엇이고 왜 중요한지"까지.
- 구체적인 명령어·설정·수치가 있으면 그대로 적는다.

## 막혔던 점과 해결
- (있었다면) 겪은 문제와 어떻게 풀었는지. 독자에게 가장 유용한 부분.

## 메모/생각
- (선택) 느낀 점, 다음에 해볼 것.
```

## 톤과 범위

- 메모 본문은 한국어로, 담백하고 명료하게.
- 이 블로그는 한국어 기술 블로그(**Physical AI 드림팀**, ditis3g.github.io)다.
  기술·학습 내용을 독자가 따라올 수 있게 적는다.
- 민감 정보(비밀번호, 토큰, 개인정보)는 메모에 넣지 마라. 블로그는 공개된다.
